### 대주제 1: Claude Skills란 무엇인가 — AI에게 주는 SOP 업무 매뉴얼

**[주제명]**: Claude Skills의 개념과 구조 — "AI판 N8N 워크플로우"

**[내용 정리]**:

#### 소주제 1.1: Skills의 핵심 비유 — 신입사원 업무 매뉴얼과 N8N 워크플로우

구시는 Claude Skills를 두 가지 강력한 비유로 설명한다.

**첫 번째 비유: 신입사원 업무 매뉴얼**

회사에 신입사원이 들어왔을 때 구두로만 "이건 이렇게 처리하고 저건 저렇게 처리해"라고 말하면, 신입사원이 그 지시를 정확히 따르는 데 한계가 생긴다. 지시가 추상적이면 결과물의 품질이 들쭉날쭉해지기 때문이다. 그래서 성숙한 조직일수록 **업무 매뉴얼(SOP, Standard Operating Procedure)**을 문서화하여 제공한다. 그 매뉴얼이 있으면 신입사원도 온보딩 후 일관된 결과물을 낼 수 있다.

Claude라는 AI에게도 동일한 논리가 적용된다. 매번 새 프롬프트를 작성하는 것은 구두 지시와 다를 바 없다. 반면 Skills는 "이 업무를 이 순서로, 이 포맷으로, 이 체크리스트에 따라 처리하라"는 문서화된 업무 매뉴얼을 Claude에게 미리 제공하는 것이다.

**두 번째 비유: N8N 자동화 워크플로우**

N8N(또는 Make, Zapier 같은 노코드 자동화 툴)에서는 보통 **트리거 → 노드(액션 1, 2, ...) → 최종 아웃풋** 구조로 워크플로우가 짜인다. Skills의 구조도 이와 완전히 대응된다.

- **Description(설명란)** = 트리거. "견적서를 만들어줘"라는 문장이 들어오면 해당 스킬이 자동 실행된다.
- **skill.md 본문** = 노드들. 실제 업무 절차, 아웃풋 포맷, 체크리스트가 여기 정의된다.
- **첨부 문서·스크립트** = 추가 액션 리소스. 필요할 때마다 로드되는 참고 문서, 코드 등이다.

이 비유는 단순히 이해를 돕기 위한 것만이 아니라, 실제 구현 원리를 정확히 반영한다. N8N으로 할 수 있는 것을 Claude Skills로도 동일하게 구현할 수 있다는 것이 구시의 핵심 주장이다.

#### 소주제 1.2: 3계층(Three-Layer) 아키텍처 — 토큰 효율의 비밀

Claude Skills는 세 가지 계층으로 설계되어 있으며, 이 구조가 토큰 효율과 일관성을 동시에 달성하는 핵심이다.

**1계층 — 이름(Name) + 설명(Description)**
항상 시스템에 로딩된다. 매우 가볍다(영상에서는 예시로 약 80토큰). Claude는 이 설명만 보고 "어떤 요청이 들어왔을 때 이 스킬을 트리거할지"를 판단한다. 수십 개의 스킬을 프로젝트에 등록해도 1계층만 항상 메모리에 올려두기 때문에 전체 컨텍스트 비용이 거의 늘지 않는다.

**2계층 — skill.md 본문**
트리거가 발생할 때만 로드된다. 실제 업무 지침(프로세스 단계, 아웃풋 포맷, 체크리스트)이 담긴 메인 문서다. 복잡하고 길어도 평소에는 로딩되지 않으므로 불필요한 토큰 낭비가 없다.

**3계층 — 참고 문서·코드 스크립트**
업무 진행 중 필요할 때만 로드된다. 예: PDF 출력용 파이썬 스크립트, 브랜드 디자인 가이드라인 문서, 특정 서식 템플릿 등. 이 계층 덕분에 '실행 가능한 코드'도 스킬 안에 포함할 수 있어 단순 텍스트 응답 이상의 결과물을 만들 수 있다.

이 3계층 설계의 핵심 가치는 **"평시엔 가볍게, 필요할 때만 무겁게"**다. 다양한 스킬을 동시에 등록해 두어도 프로젝트 성능에 부담을 주지 않으면서, 실행 시에는 풍부한 컨텍스트를 활용할 수 있다.

#### 소주제 1.3: 프롬프트와 Skills의 세 가지 핵심 차이

구시는 "그냥 프롬프트 잘 쓰는 것과 뭐가 다르냐"는 잠재적 의문에 대해 세 가지 축으로 답한다.

**① 재사용성**: 프롬프트는 매번 복사-붙여넣기가 필요하다. Skills는 한 번 만들어두면 트리거 키워드(또는 슬래시 커맨드)만으로 바로 실행된다. 복잡한 지시사항을 한 번만 만들면 그 이후로는 자동이다.

**② 일관성**: 프롬프트가 길고 복잡해질수록 AI가 모든 조건을 빠짐없이 지키는 데 한계가 생긴다. Skills는 체크리스트와 코드 스크립트를 조합하여 구조적으로 일관성을 강제한다. 특히 코드 스크립트를 직접 실행하여 결과물을 생성하는 방식은 '매번 조금씩 달라지는' 창의적(?) 변동성을 제거한다.

**③ 확장성**: 프롬프트는 텍스트·이미지 인풋에 머문다. Skills는 코드 스크립트, 3rd-party 문서, 서브 에이전트까지 통합할 수 있어 훨씬 복잡한 자동화 파이프라인을 구성할 수 있다.

한 문장으로 요약하면: 프롬프트가 N8N의 "단일 노드 설정"이라면, Skills는 N8N의 "전체 워크플로우 자체"다.

이 스킬은 세 가지 레이어로 구조가 나눠져 있는데요.

필요한 경우에 따라서만 이 두 번째, 세 번째 레이어를 로드해서 작업을 하기 때문에
토큰을 효율적으로 사용하면서도 일관된 결과물을 생성할 수가 있습니다.

자, 이렇게까지 들으시면 "어, 그냥 프롬프트 엔지니어링을 잘하는 것과 비슷한 거 아니냐"라고
생각하실 수도 있는데요. 프롬프트와는 세 가지 핵심적인 차이점이 있습니다.

먼저 첫 번째로 스킬스는 재사용성이 뛰어난데요.

클로즈 스킬스는 원하는 형태의 워크플로우를 항상 지정을 해 놓고
그거를 트리거만 해 주면 되기 때문에
훨씬 복잡한 지시 사항을 한 번만 만들면 자동으로 실행을 시켜 줄 수 있다는 장점이 있습니다.

그리고 제 생각에 가장 중요한 차이점은 일관성이 굉장히 뛰어나다는 건데요.
우리가 프롬프트는 복잡하고 길게 요청을 할수록
사실 AI가 항상 그 지침을 하나도 빠짐없이 일관된 결과물을 생성해 주는 데 한계가 있는 편입니다.
반면에 스킬스는 아까 보신 것처럼 세 가지 레이어를 활용하고
그 과정에서 포맷이나 체크리스트 같은 거를 참고 문서로도 첨부를 해 줄 수 있지만
그것뿐만 아니라 코드 스크립트를 제작해서 그냥 코드를 실행해서 결과물을 반환해 줄 수 있기 때문에
일관된 결과물을 만드는 데 굉장히 특화되어 있는 기능이라고 볼 수 있습니다.

또 세 번째로는 확장성이 훨씬 뛰어난데요.

#### 소주제 2.1: 환경 설정 — VS Code + Claude Code 터미널 세팅

구시는 실습 환경으로 VS Code + Claude Code 터미널 조합을 선택한다. 이 영상의 대상 시청자는 이미 VS Code, Node.js, Claude Code를 설치한 상태라고 가정하며, 아직 설치하지 않은 분들에게는 이전 영상을 먼저 참고하도록 안내한다.

실습 흐름은 다음과 같다:
1. VS Code를 열고 작업 폴더(`skills/`)를 생성 후 오픈한다.
2. 터미널을 열고(메뉴 → Terminal → New Terminal), 화면 우측으로 배치한다.
3. `claude --dangerously-skip-permissions` 옵션으로 Claude Code를 실행한다. 이 옵션은 매 명령마다 권한 확인 팝업 없이 자동 실행되도록 하는 설정으로, 빠른 작업 흐름을 원할 때 유용하다(물론 보안 리스크를 충분히 인지한 상태에서 사용해야 한다).

**두 가지 스킬 제작 방법**:
- 직접 `skill.md` 파일을 작성하는 방법 — 숙련자에게 적합.
- **Skill Creator 플러그인을 이용하는 방법** — 오늘의 주요 방법. 스킬을 만들어주는 스킬 자체를 활용하는 것이다.

Skill Creator는 Claude Code 공식 마켓플레이스에서 설치 가능하다. `/plugin` 명령으로 마켓플레이스에 접근하고, "skill creator"를 검색하여 설치한다. 이미 설치된 경우 `Installed` 탭에서 확인할 수 있다. 설치 후 `/exit`으로 나갔다가 재진입하면 `/skill-creator` 슬래시 커맨드가 활성화된다.

#### 소주제 2.2: 스킬 제작 요청 — 구체적 프롬프트와 레퍼런스 제공

스킬을 잘 만들기 위해 구시는 두 가지 사전 준비를 한다.

**① 레퍼런스 PDF 첨부**: `reference/` 폴더에 인보이스 템플릿 PDF를 넣는다. 견적서의 레이아웃, 포함 항목, 구성 방식을 Claude가 시각적으로 파악하게 하기 위해서다.

**② 구체적인 스킬 요청 프롬프트**:
- 고객사명, 프로젝트 내용, 업무 항목과 단가를 입력받아 견적 초안 생성
- 아웃풋은 PDF 파일로 생성
- 자체 검증 체크리스트 포함
- 특정 멘트(예: "견적서 만들어줘")에 자동 트리거
- 인보이스 템플릿 PDF 참고
- **테스트도 함께 진행해달라는 요청 포함**

이 마지막 항목이 핵심이다. 단순히 스킬을 만들어 달라는 게 아니라, 만든 후 AB 테스트까지 바로 실행해달라고 요청함으로써 제작→검증→개선의 전체 사이클을 한 번에 돌리는 것이다.

**스킬 제작 결과물 (3계층 구조)**:

- **1계층**: `skill.md` 최상단에 Name과 Description 기재. 약 80토큰.
- **2계층**: skill.md 본문. 1단계(정보 수집) → 2단계(JSON 데이터 준비) → 3단계(PDF 생성) → 4단계(검증 체크리스트 확인) → 5단계(파일 경로 안내) 순서로 워크플로우 정의.
- **3계층**: PDF 생성용 파이썬 스크립트 자동 생성. 이 스크립트가 실제 PDF 파일을 만드는 실행 코드 역할을 한다.

#### 소주제 3.1: 피드백 제출 — 브랜드 가이드라인 문서 추가

1차 테스트 결과를 보면 스킬 적용 버전이 확실히 일관성 면에서 우수하지만, 디자인 측면에서 개선 여지가 있다. 구시는 여기서 피드백 기능을 활용한다.

**브랜드 디자인 가이드라인 문서 추가**:
`reference/` 폴더에 브랜드 가이드라인 `.md` 문서를 추가한다. 이 문서에는 메인 컬러(주황색), 서브 컬러, 폰트, 텍스트 스타일 등 브랜드 아이덴티티 요소들이 정의되어 있다.

**피드백 내용**:
- 브랜드 디자인 가이드라인 MD 문서 기준으로 PDF 디자인 수정
- 색상과 폰트를 가이드라인에 맞춰 적용
- 헤더/푸터에 회사 정보 포함
- 전체적으로 전문적이고 깔끔한 느낌으로 개선

이 피드백은 `feedback.json` 파일로 로컬에 자동 저장된다. 이후 `피드백을 제공했으니 스킬을 업데이트해줘`라는 요청 하나로 Claude Code가 해당 JSON 파일을 자동 검색·로드하여 스킬을 개선한다.

#### 소주제 3.2: 2차 이터레이션 및 벤치마크 결과

피드백을 반영한 스킬 개선 후 2차 테스트가 자동으로 실행된다. 결과는 뷰어에서 "Previous Output"과 "Current Output"을 나란히 비교할 수 있어 개선 전후를 직관적으로 파악할 수 있다.

**2차 결과 하이라이트**:
- 브랜드 가이드라인에 지정한 주황색이 헤더 등에 반영됨
- 회사 정보가 헤더/푸터에 올바르게 배치됨
- 전반적인 레이아웃이 더 전문적으로 개선됨
- 처리 시간 및 토큰 효율성 수치가 벤치마크에 정량적으로 표시됨

이 이터레이션 루프의 진짜 가치는 **"제작 → 테스트 → 피드백 → 개선"을 Claude Code 안에서 단일 워크플로우로 완결**할 수 있다는 점이다. 외부 도구 없이 스킬의 품질을 수치로 측정하고 반복 개선할 수 있다.

#### 소주제 4.2: 로컬 마켓플레이스 생성 — 플러그인 패키징

단순 복사-붙여넣기를 넘어, 스킬을 마켓플레이스에 배포 가능한 플러그인 형태로 패키징하는 방법을 소개한다.

**참고할 공식 문서 활용**: 마켓플레이스 생성·배포 방법에 대한 공식 Claude 문서 링크를 프롬프트에 그대로 붙여 넣어 Claude Code에 전달한다. 구시는 이 방법을 일반적인 원칙으로 강조한다 — "뭔가 설치해야 할 때 그냥 요청하기보다는 공식 문서 링크를 함께 넣어주면 공식 형태에 맞게 더 정확히 만들어 준다."

**생성되는 파일 구조**:
1. `claude-plugin/` 폴더: 스킬 파일 포함
2. `.claude/plugins/marketplace.json`: 마켓플레이스 메타데이터

플러그인을 로컬 마켓플레이스에서 `Add Marketplace → 로컬 경로 입력`으로 추가하면, `/plugin` 메뉴에서 "Citizen Dev Marketplace"가 나타나고 여기서 Quote Generator를 선택·설치할 수 있다.

#### 소주제 4.3: Claude Desktop(Claude.app) 코워크에서 ZIP 플러그인 설치

Claude Code(터미널)뿐 아니라 Claude Desktop 앱의 "Claude.app Work" 기능에서도 동일한 스킬을 사용하고 싶다면 별도 설치가 필요하다.

**Claude Desktop 설치 절차**:
1. Claude Desktop 앱 실행
2. Cowork 탭에서 작업 폴더 설정
3. `/plugin` → Personal → `+` → Upload Plugin
4. ZIP 파일 업로드

만약 GitHub에 마켓플레이스 저장소를 올려두었다면 "GitHub에서 추가" 방식으로도 설치 가능하다. 로컬 전용이라면 ZIP 업로드로 충분하다.

**최종 활용 시연**: Claude Desktop의 코워크에서 "DEF 스타트업에서 자동화 프로젝트 견적을 요청했다. AI 자동화권이고 단가 100만 원 기준으로 견적서를 생성해줘"라는 단순한 요청만으로 견적서 PDF가 깔끔하게 생성된다. 스킬이 없었다면 레이아웃, 디자인, 필수 항목, 체크리스트 등을 매번 직접 지시해야 했을 것이다.

## [이번 화 전체 내용 요약 정리]

이 영상은 Claude Code의 **Skills(스킬스) 기능**을 처음부터 완전히 다루는 실전 튜토리얼이다. 영상의 전체 흐름은 크게 네 단계 — 개념 이해 → 스킬 제작 → 이터레이션 개선 → 플러그인 배포 — 로 구성되며, 각 단계가 긴밀하게 연결되어 하나의 완성된 워크플로우를 형성한다.

**핵심 개념**부터 살펴보면, Skills는 Claude에게 부여하는 SOP(표준 업무 절차) 매뉴얼이다. 신입사원에게 구두 지시 대신 문서화된 매뉴얼을 주는 것과 같은 원리로, Claude가 반복 업무를 매번 일관되게 처리할 수 있도록 돕는다. 구조적으로는 N8N의 자동화 워크플로우와 동형(同形)이다 — Description이 트리거, skill.md가 노드 체인, 첨부 문서·스크립트가 추가 리소스에 해당한다.

**3계층 아키텍처**는 이 기능의 기술적 핵심이다. 1계층(이름+설명, ~80토큰)만 항상 로딩되고 나머지는 필요할 때만 로드되기 때문에, 수십 개의 스킬을 등록해도 평시 토큰 비용이 거의 없다. 동시에 실행 시에는 코드 스크립트, 브랜드 가이드라인, 서브 에이전트 등 풍부한 리소스를 모두 활용할 수 있다.

**프롬프트 대비 Skills의 세 가지 우위**는 재사용성(트리거 한 번으로 복잡한 워크플로우 실행), 일관성(체크리스트·코드 스크립트로 구조적 품질 보장), 확장성(코드·문서·서브에이전트 통합)이다. AB 테스트에서는 이를 수치로 입증했다: 처리 시간 34초 vs. 147초, 토큰 사용량도 대폭 절감, 결과물 일관성에서도 압도적 차이를 보였다.

**실습 과정**에서는 Skill Creator 플러그인을 활용하여 견적서 자동 생성 스킬을 만들었다. 인보이스 템플릿 PDF를 레퍼런스로 제공하고, 구체적인 요구사항(트리거 조건, 아웃풋 포맷, 체크리스트, PDF 생성)을 명시한 요청으로 스킬이 제작되었다. 스킬 제작 즉시 AB 테스트가 자동 실행되어 성능을 정량적으로 확인할 수 있었다.

**이터레이션 루프**를 통해 1차 테스트에서 발견된 디자인 문제를 브랜드 가이드라인 문서 추가와 피드백 제출로 해결했다. Claude Code가 feedback.json을 자동 검색·로드하여 스킬을 개선하고 2차 테스트를 재실행하는 과정은, "제작 → 검증 → 피드백 → 개선"의 전체 사이클이 단일 환경 안에서 완결됨을 보여준다.

마지막으로 **플러그인 배포**를 통해 완성된 스킬을 다른 프로젝트나 Claude Desktop에서도 사용할 수 있게 패키징하는 방법을 실습했다. 로컬 마켓플레이스 생성, ZIP 패키징, Claude Desktop 업로드의 세 단계로 어디서든 동일한 스킬을 즉시 활용할 수 있는 체계를 갖추었다.

전체 영상의 메시지는 명확하다: **반복적으로 수행하는 업무가 있다면 그 업무를 Skills로 정의하라.** 처음 한 번의 투자(스킬 제작)로 이후의 모든 실행에서 일관성·속도·품질을 동시에 얻을 수 있다. Skills는 프롬프트 엔지니어링의 다음 단계, AI 자동화의 실질적 출발점이다.
