## 2. 기획 의도

### 문제 인식

기존 구조의 한계:
1. **fetch와 digest가 섞여 있었다** — `--fetch-only` / `--digest-only` 분리는 됐지만, "어떤 fetch에서 나온 결과인가"를 추적할 방법이 없었다.
2. **pending.json이 너무 많은 역할을 했다** — 현재 배치 상태 + fetch 완료 기록 + 이력이 혼재. pending이 비워지면 이전 배치 정보가 소실됐다.
3. **여러 번 fetch가 예상되는데 묶을 단위가 없었다** — popular fetch, range fetch, keyword fetch가 언젠가 중복 실행될 것. 어느 fetch가 어느 영상을 커버했는지 추적 불가.
4. **인기순 정렬이 digest 계획에 반영되지 않았다** — registry가 단순 dict이므로 "인기 3위부터 10위까지 처리해줘" 같은 순서 기반 요청이 불가능했다.

### 설계 목표

```
fetch 실행 하나 = "run" 단위로 묶기
run.video_ids 순서가 곧 digest 우선순위
pending.json = 현재 진행 중인 digest 계획만 (완료 즉시 archived)
registry = 영속 fetch 원장 (상태 + digest 이력 누적)
```

## 3. 신규 파일

### `src/fetch_runs.py`

Fetch 실행 단위를 `cache/fetch_runs.json`에 누적 관리.

```python
make_run_id(kind)    # '20260425T182416_popular'
save_run(run_id, kind, channel_url, params, video_ids)
get_run(run_id)
latest_run(kind=None)
load_all_runs()
```

**핵심 설계 포인트**:
- run_id는 `YYYYMMDDTHHMMSS_{kind}` 형태 — 타임스탬프 자체가 정렬 키
- `video_ids`의 순서가 fetch 시점의 "의도된 순서"를 보존 (popular → 인기순, range → 날짜역순)
- 같은 run_id로 재호출 시 video_ids 덮어쓰기 (idempotent)

---

### `src/digest_archive.py`

Digest 완료 후 `pending.json`을 `cache/digested/{ts}.json`으로 이동.

```python
archive_pending(pending_path)   # → 이동 후 경로 반환
load_archive(name)
list_archives()
```

**설계 의도**: pending.json은 "현재 진행 중인 작업"만 담아야 한다. 완료된 배치는 타임스탬프 파일로 보존해 이력 추적은 가능하되 현재 상태에는 영향을 주지 않는다.

---

### `src/plan.py`

Registry + Fetch Run 데이터를 읽어 새로운 `pending.json`을 작성.

```python
plan_from_registry(
    cfg,
    from_run='latest',        # 'latest' 또는 run_id
    from_run_kind=None,       # 'popular' | 'range' | ...
    top=None,                 # 상위 N개만
    mode='heavy',
    skip_already=True,        # 같은 mode로 이미 처리된 영상 제외
)
```

**핵심 설계 포인트**:
- run의 `video_ids` 순서를 그대로 사용 → popular run이면 인기순 그대로
- `skip_already`: 같은 영상을 다른 mode로 재처리하는 건 허용, 같은 mode 중복만 제외
- `subdir='인기'` — run kind가 'popular'면 자동 세팅 → 출력 폴더 자동 분기

## 4. 기존 파일 수정

### `src/registry.py` — run_id 지원

```python
# 이전
def add_queued(video, registry_path=REGISTRY_PATH):

# 이후
def add_queued(video, registry_path=REGISTRY_PATH, run_id=None):
```

이미 등록된 영상이면 status는 유지하고 `fetch_runs` 배열에만 run_id 누적. 같은 영상이 여러 run에 속할 수 있다 (popular에도 있고 range에도 있는 경우).

---

### `main.py` — 대규모 수정

**1. 신규 import**
```python
from src import registry, fetch_runs
```

**2. `_build_entry(video, languages, run_id=None)`**
- `registry.add_queued(video, run_id=run_id)` 전달

**3. `_process_video_list()` — kind/run_params/subdir 추가**
```python
def _process_video_list(videos, cfg, ..., kind=None, run_params=None, subdir=''):
    run_id = fetch_runs.make_run_id(kind) if kind else None
    ...
    fetch_runs.save_run(run_id, kind, channel_url, params, video_ids)
    manifest = _save_manifest(..., subdir=subdir)
```
fetch 완료 시점에 run을 기록한다. `attempted_ids` = 실제 fetch 시도한 영상들 (skip_processed로 제외된 것 포함하지 않음).

**4. 5개 run_* 함수에 kind 연결**

| 함수 | kind | run_params |
|------|------|------------|
| `run_single` | `'single'` | `{'url': video_url}` |
| `run_latest` | `'latest'` | `{}` |
| `run_range` | `'range'` | `{'start', 'end', 'skip_processed'}` |
| `run_keyword` | `'keyword'` | `{'keyword', 'start', 'end', 'skip_processed'}` |
| `run_popular` | `'popular'` | `{'top', 'skip_processed'}` |

**5. `_save_manifest()` — subdir 필드 추가**

**6. 3개 처리 함수 — subdir 전달**
```python
subdir = manifest.get('subdir', '')
...
filepath = save_markdown(..., subdir=subdir)
```

**7. argparse — 3개 플래그 추가**
```
--digest-from-registry    fetch run 기반 plan → digest
--from-run RUN_ID         대상 run ('latest' 기본)
--from-run-kind KIND      latest와 함께 쓸 kind 필터
```

**8. `main()` dispatch — `--digest-from-registry` 핸들러**
```python
if args.digest_from_registry:
    manifest = plan_from_registry(cfg, from_run, from_run_kind, top, mode)
    run_digest_only(cfg, manifest, ...)
    return
```

### `src/storage.py` — subdir 파라미터

```python
def save_markdown(content, channel_name, date, title, base_dir='./output', prefix='', subdir=''):
    channel_dir = os.path.join(base_dir, sanitize_dirname(channel_name))
    if subdir:
        channel_dir = os.path.join(channel_dir, subdir)
```

`mark_processed`는 subdir 없이 루트 채널 디렉터리를 쓴다 — `.processed.json`은 항상 채널 루트에 존재해야 skip_processed 로직이 정상 작동하기 때문.
